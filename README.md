# Next.js ERP Assignment

## 1. 프로젝트 소개

Next.js 기반으로 간단한 ERP 시스템을 구현하는 과제 프로젝트입니다.  
상품 및 거래처 관리, 구매/판매 프로세스, 대시보드 기능을 구현하는 것을 목표로 합니다.

## 2. 기술 스택

#### 프레임워크: Next.js (App Router)

#### 라이브러리: React

#### 언어: TypeScript

- 타입 안정성을 확보하고 코드 가독성을 높이기 위해 사용합니다.

#### 스타일링: Tailwind CSS

- 관리자 페이지 UI를 빠르게 구성하기 위해 사용합니다.

#### 인증: Cookie기반, Next.js Middleware

#### 런타임 환경: Node.js

#### 데이터베이스: SQLite, Prisma ORM

- 로컬 환경에서 쉽게 실행 가능하고 별도 DB 서버 설치가 필요 없기 때문에 선택하였습니다.
- Next.js 환경에서 데이터베이스 모델링과 CRUD 작업을 단순화하기 위해 사용합니다.
- Prisma를 통해 타입 데이터베이스 접근을 안전하게 합니다.

## 3. 프로젝트 구조

```text
/app
  /admin
    /dashboard         // 대시보드 요약
    /partners          // 거래처 CRUD
    /products          // 상품 CRUD
    /purchase-orders   // 구매 처리
    /sales-orders      // 판매 처리
  /api
    /auth
      /login             // 로그인 API
      /logout            // 로그아웃 API
  /login               // 로그인
  /middleware.ts       // 인증 검증 미들웨어
```

비로그인 사용자의 관리자 접근을 방지하기 위해
관리자 영역을 /admin 경로로 통일하고 middleware에서 인증 검사를 수행합니다.

## 4. 주요 기능

### - 관리자 로그인

#### 개발용 계정

```text
ID: admin
Password: admin123
```

#### 인증 흐름

1. 로그인 페이지에서 관리자 계정 입력
2. /api/login API 호출
3. 서버에서 계정 검증
4. 성공 시 인증 쿠키 생성
5. middleware에서 보호 경로 접근 시 쿠키 검사
6. 인증 실패 시 로그인 페이지로 리다이렉트

#### 보호 경로

다음 경로는 인증이 필요한 관리자 영역입니다.

```text
/admin/*
```

### - 상품 관리 (CRUD)

- 상품 등록

- 상품 목록 조회

- 상품 검색 (상품코드 / 상품명 기준)

- 상품 정보 수정

- 상품 상세 조회

- 상품 비활성화 처리

상품 삭제는 기존 거래 이력과의 데이터 정합성을 유지하기 위해
실제 데이터 삭제 대신 판매여부(isSale) 값을 변경하는 소프트 삭제 방식으로 설계하였습니다.

### - 거래처 관리 (CRUD)

- 거래처 등록

- 거래처 목록 조회

- 거래처 검색 (거래처명 기준)

- 거래처 정보 수정

- 거래처 상세 조회

- 거래처 비활성화 처리

### - 발주 생성 및 입고 처리

- 입고 처리

- 재고 변동 이력 기록

- 현재 재고 자동 반영

### - 주문 생성 및 출고 처리

- 출고 처리

- 재고 변동 이력 기록

- 현재 재고 자동 반영

### - 대시보드 요약 정보

- 상품 수

- 거래처 수

- 재고 현황

- 주요 운영 지표 요약

## 5. 데이터베이스 구조

본 프로젝트는 ERP 시스템의 기본 데이터 구조를 단순화하여 설계하였습니다.

### Product (상품)

상품의 기본 정보를 관리하는 마스터 데이터 테이블입니다.

#### 주요 컬럼

- 상품코드

- 상품명

- 단가

- 안전 재고

- 현재 재고

- 생성일

- 판매여부

안전 재고는 재고 부족 상태를 판단하기 위한 기준값으로 사용됩니다.

### Partner (거래처)

거래처 정보를 관리하는 테이블입니다.

ERP 시스템에서는 고객과 공급사를 모두 거래처로 관리하는 경우가 많기 때문에
본 프로젝트에서는 하나의 테이블에서 거래처 유형으로 구분하도록 설계하였습니다.

#### 거래처 유형

CUSTOMER (고객)

SUPPLIER (공급사)

#### 주요 컬럼

- 거래처명

- 거래처 유형

- 연락처

- 이메일

- 계약 상태

- 생성일

### StockMovement (재고 변동 기록)

상품의 재고 변동을 기록하는 테이블입니다.

재고는 입고 및 출고 이벤트에 의해 변경되며
모든 재고 변경 이력은 해당 테이블에 기록됩니다.

#### 재고 변동 유형

IN (입고)

OUT (출고)

ADJUST (재고 조정)

이를 통해 재고 변경 이력을 추적할 수 있습니다.

#### 주요 컬럼

- 변동 수량

- 변동 유형

- 사유

- 발생일

- 변동 기록 아이디

변동 기록 아이디는 주문 / 발주 테이블의 기본 키를 참조하는 polymorphic reference 구조입니다. 어떤 곳에서 발생한 재고 변동인지 바로 알기 위한 컬럼입니다.

### PurchaseOrders (발주)

공급사로부터 상품을 구매하기 위한 발주 정보를 관리하는 테이블입니다.

발주 생성 시점에는 재고가 변경되지 않으며,
실제 입고 처리 시 재고가 증가하고 재고 변동 이력이 기록됩니다.

#### 발주 상태 유형

REQUESTED (발주 접수)

RECEIVED (입고 완료)

CANCELED (발주 취소)

#### 주요 컬럼

- 거래처 아이디 (공급사)

- 상품 아이디

- 발주 수량

- 발주 상태

- 생성일

발주는 입고 완료 이후에는 취소할 수 없도록 설계하였습니다.
이는 재고 반영 이후 단순 취소가 아닌 별도 재고 조정이 필요하기 때문입니다.

### SalesOrders (주문)

고객의 상품 주문 정보를 관리하는 테이블입니다.

주문 생성 시점에는 재고가 변경되지 않으며,
출고 처리 시 재고가 감소하고 재고 변동 이력이 기록됩니다.

#### 주문 상태

RECEIVED (주문 접수)

SHIPPED (출고 완료)

CANCELED (주문 취소)

#### 주요 컬럼

- 거래처 아이디 (고객)

- 상품 아이디

- 주문 수량

- 주문 상태

- 생성일

주문은 출고 완료 이후에는 취소할 수 없도록 설계하였습니다.
이미 재고가 차감된 상태이므로 단순 취소가 아닌 반품 또는 재고 조정 프로세스가 필요하기 때문입니다.

### 테이블 관계 구조

본 프로젝트의 데이터베이스는 상품, 거래처, 주문, 발주, 재고 변동 기록 간의 관계를 기반으로 구성되었습니다.

하나의 거래처는 여러 건의 발주 및 주문을 생성할 수 있으며,
하나의 상품 또한 여러 건의 발주 및 주문에 포함될 수 있습니다.

따라서 발주 및 주문 테이블은 각각 거래처와 상품을 참조하는 구조로 설계하였습니다.

또한 하나의 상품은 여러 건의 재고 변동 기록을 가질 수 있도록 구성하여
상품 단위의 재고 이력을 추적할 수 있도록 하였습니다.

이를 통해 다음과 같은 관계 구조를 가집니다.

거래처(Partner) 1 : N 발주(PurchaseOrders)

거래처(Partner) 1 : N 주문(SalesOrders)

상품(Product) 1 : N 발주(PurchaseOrders)

상품(Product) 1 : N 주문(SalesOrders)

상품(Product) 1 : N 재고 변동 기록(StockMovement)

이와 같은 구조를 통해 ERP 시스템의 기본적인 구매 및 판매 흐름을 단순화하여 표현하였습니다.

### 재고 처리 흐름

재고는 주문 및 발주 처리 과정에서 변경되며,
모든 재고 변경 이력은 재고 변동 기록 테이블에 저장됩니다.

#### 발주 및 입고 처리

발주 생성 시점에는 재고가 변경되지 않습니다.

실제 입고 처리 시 상품의 현재 재고가 증가합니다.

동시에 재고 변동 기록 테이블에 입고(IN) 유형의 이력이 저장됩니다.

발주 상태는 입고 완료(RECEIVED) 상태로 변경됩니다.

#### 주문 및 출고 처리

주문 생성 시점에는 재고가 변경되지 않습니다.

출고 처리 시 상품의 현재 재고가 감소합니다.

동시에 재고 변동 기록 테이블에 출고(OUT) 유형의 이력이 저장됩니다.

주문 상태는 출고 완료(SHIPPED) 상태로 변경됩니다.

#### 재고 조정 처리

재고 오류, 파손, 수량 정정 등의 상황에서는 재고 조정 기능을 통해
재고 변동 기록 테이블에 ADJUST 유형의 이력을 남기도록 설계하였습니다.

이를 통해 재고의 모든 변경 내역을 추적할 수 있으며,
현재 재고는 상품 테이블의 현재 재고 컬럼을 기준으로 조회할 수 있습니다.

## 시스템 공통 처리 정책

본 프로젝트에서는 데이터 무결성과 안정적인 API 응답 처리를 위해
다음과 같은 공통 정책을 적용하였습니다.

### 입력값 검증

모든 등록 및 수정 API 요청에 대해 서버에서 필수값 검증과
데이터 타입 검증을 수행하도록 설계하였습니다.

이를 통해 잘못된 요청 데이터가 데이터베이스에 저장되는 것을 방지하였습니다.

### 상태코드 처리

API 응답 시 상황에 따라 적절한 HTTP 상태코드를 반환하도록 구현하였습니다.

- 200: 조회 및 수정 성공

- 201: 등록 성공

- 400: 잘못된 요청 데이터

- 404: 존재하지 않는 데이터

- 409: 데이터 충돌

- 500: 서버 내부 오류

### 검색 처리

목록 조회 시 검색어 기반 필터링 기능을 제공하며,
검색어의 공백 입력으로 인한 오류를 방지하기 위해 서버에서 문자열 trim 처리를 수행하도록 설계하였습니다.
