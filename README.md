# Next.js ERP Assignment

## 1. 프로젝트 소개

Next.js(App Router) 기반으로 구현한 경량 ERP 시스템입니다. 
상품 및 거래처 관리, 구매/판매 프로세스, 재고 추적 및 대시보드 시각화까지 핵심적인 비즈니스 로직을 포함하고 있습니다.

## 2. 기술 스택

#### - 프레임워크: Next.js (App Router)

#### - 라이브러리: React

#### - 언어: TypeScript

- 타입 안정성을 확보하고 코드 가독성을 높이기 위해 사용합니다.

#### - 스타일링: Tailwind CSS

- 관리자 페이지 UI를 빠르게 구성하기 위해 사용합니다.

#### - 인증: Cookie기반 인증 + Next.js Middleware

#### - 런타임 환경: Node.js

#### - 데이터베이스: SQLite

- 로컬 환경에서 쉽게 실행 가능하고 별도 DB 서버 설치가 필요 없기 때문에 선택하였습니다.

#### - ORM: Prisma

- Next.js 환경에서 데이터베이스 모델링과 CRUD 작업을 단순화하기 위해 사용합니다.

## 3. 프로젝트 구조

```text
/app
  /admin
    /dashboard         // 대시보드 요약
    /partners          // 거래처 CRUD
    /products          // 상품 CRUD
    /purchase-orders   // 구매 처리
    /sales-orders      // 판매 처리
    /stocks            // 재고 관리
  /api
    /auth              // 인증
    /partners          // 거래처
    /products          // 상품
    /purchase-orders   // 구매
    /sales-orders      // 판매
    /stocks            // 재고
  /login               // 로그인
/middleware.ts         // 인증 검증 미들웨어
```

## 4. 구현 범위

### 4-1 관리자 로그인 (인증)

#### 개발용 계정

```text
ID: admin
Password: admin123
```

#### 인증 흐름

1. 로그인 페이지에서 관리자 계정 입력
2. /api/login API 호출
3. 서버에서 계정 검증
4. 성공 시 인증 쿠키 생성
5. middleware에서 보호 경로 접근 시 쿠키 검사
6. 인증 실패 시 로그인 페이지로 리다이렉트

#### 보호 경로

비로그인 사용자의 관리자 접근을 방지하기 위해
관리자 영역을 /admin 경로로 통일하고 middleware에서 인증 검사를 수행합니다.

### 4-2 마스터 데이터 관리

#### 상품 관리

- 상품 등록 / 수정 / 상세 조회 / 목록 조회

- 상품코드 및 상품명 기준 검색

- 데이터 정합성을 위해 물리 삭제 대신 **비활성화 처리(Soft Delete)** 적용

#### 거래처 관리

- 거래처 등록 / 수정 / 상세 조회 / 목록 조회 / 검색

- 고객 / 공급사 유형을 하나의 거래처 도메인으로 통합 관리

### 4-3 발주 생성 및 입고 처리

- 발주서 생성

- 발주 목록 조회

- 발주 상태 변경

- 발주 상태 이력 조회

- 입고 완료 처리

- 입고 시 재고 자동 증가 및 재고 변동 기록 생성

발주는 공급사를 대상으로 생성되며,
하나의 발주서에 여러 상품을 포함할 수 있도록 구현하였습니다.

발주 생성 시점에는 재고가 변경되지 않고,
**입고 완료 처리 시점**에만 상품 재고가 증가합니다.

또한 발주 상태가 변경될 때마다 상태 이력을 별도 테이블에 기록하여
발주 진행 흐름을 추적할 수 있도록 설계하였습니다.

요구사항의 상태 정의(초안/확정/완료)는
ERP 일반 용어 기준으로 재정의하여 사용하였습니다.

### 4-4 주문 생성 및 출고 처리

- 주문서 생성

- 주문 상태 관리

- 주문 상태 변경 이력 기록

- 출고 완료 처리

- 출고 시 재고 자동 감소 및 재고 변동 기록 생성

- 재고 부족 검증

주문은 고객사를 대상으로 생성되며,
하나의 주문서에 여러 상품을 포함할 수 있도록 구성하였습니다.

주문 생성 시점에는 재고가 즉시 차감되지 않으며,
**출고 완료 처리 시점**에만 상품 재고가 감소하도록 설계하였습니다.

#### 재고 부족 처리 정책

주문 등록 및 출고 처리 시 현재 재고를 기준으로 수량을 검증합니다.

주문 수량이 현재 재고를 초과하는 경우 주문 등록 또는 출고 처리를 제한합니다.

주문 처리 후 재고가 안전재고 미만이 되는 경우에는 주문은 허용하되,
대시보드 및 재고 관리에서 **보충 필요 상태**를 확인할 수 있도록 구성하였습니다.

### 4-5 재고 관리

- 재고 변동 이력 목록 조회

- 재고 변동 이력 검색 (상품코드 / 상품명 기준)

- 상품 별 입고 / 출고 / 조정에 따른 재고 변동 이력 상세 조회

### 4-6 대시보드

운영자가 로그인 직후 당일 운영 현황을 파악할 수 있도록
다음 핵심 지표를 요약 표시합니다.

- 오늘 생성된 주문 건수

- 오늘 처리된 입고 건수

- 오늘 처리된 출고 건수

- 현재 재고 부족 상품 수

- 재고 부족 상품 5개

- 최근 6개월 판매량 추이 차트

## 5. 데이터베이스 구조

본 프로젝트는 ERP 시스템의 기본 데이터 구조를 단순화하여 다음과 같은 데이터 구조로 설계하였습니다.

### 5-1 주요 테이블

- Product
  상품 관리(상품 코드, 단가, 현재 재고, 안전 재고, 활성화 여부)

- Partner
  거래처 관리 (거래처 명, 거래 유형, 활성화 여부)

- PurchaseOrder / PurchaseOrderProduct
  발주 문서 Header-Detail 구조
  (거래처, 발주 상태) - (발주 문서, 상품, 수량, 단가)

- SaleOrder / SaleOrderProduct
  주문 문서 Header-Detail 구조
  (거래처, 주문 상태) - (주문 문서, 상품, 수량, 단가)

- StockMovement
  입고 / 출고 / 조정에 따른 재고 변동 기록

- PurchaseOrderStatusLog / SaleOrderStatusLog
  발주 / 주문 상태 변경 이력 관리

### 5-2 설계 특징

1. 주문 및 발주를 문서 구조인 **Header-Detail** 방식으로 설계하였습니다.

2. **상태 변경 이력**을 분리하여 업무 흐름 추적이 가능합니다.

3. 상품 삭제 대신 **비활성화 처리**로 데이터를 보존하였습니다.

4. 거래 흐름을 명확히 분리하기 위해 **발주와 주문을 별도 도메인**으로 분리하였습니다.

## 7. 실행 환경 안내

- Node.js 18 이상 환경에서 실행 가능합니다.

- SQLite기반으로 구성되어 별도의 DB 서버 설치가 필요하지 않습니다.

- 압축 파일에 DB파일이 포함되어 있어 별도의 DB 초기화 없이 실행 가능합니다.

- 프로젝트 실행에 필요한 .env 파일은 압축파일에 포함되어 있습니다.
별도의 환경 변수 설정 없이 실행 가능합니다.

1. 압축 파일 해제 후 프로젝트 폴더로 이동

2. 패키지 설치

```bash
npm install
```

3. 개발 서버 실행

```bash
npm run dev
```

4. 브라우저 접속

```text
http://localhost:3000
```

### 참고 사항

DB를 초기화하여 처음부터 테스트하려는 경우에는 Prisma 스키마 기준으로 다시 생성할 수 있습니다.

```bash
npx prisma generate
npx prisma db push
```

초기 데이터가 필요한 경우에는 seed 명령을 추가로 실행할 수 있습니다.

```bash
npx prisma db seed
```

## 8. 설계 의도 및 트레이드오프

### 8-1 설계 의도

ERP 시스템의 기본 업무 흐름을 이해하고
데이터 정합성과 운영 추적성을 확보하는 것을 목표로 설계하였습니다.

- 재고 반영 시점을 **입고 및 출고 완료 시점**으로 제한하여, 문서 생성 이후 취소나 변경이 발생할 수 있는 상황에서의 재고 오류를 최소화했습니다.

- 또한 **문서 상태 변경 이력과 재고 변동 이력**을 별도로 관리하여
업무 처리 과정과 재고 흐름을 운영자가 추적할 수 있도록 하였습니다.

### 8-2 트레이드오프

- 재고 반영 시점을 입고/출고 완료 기준으로 설계하여
재고 오류 가능성을 줄였지만 **실시간 재고 예약 기능**은 고려하지 않았습니다.

- 상품 및 거래처 삭제를 비활성화 방식으로 처리하여
거래 데이터 보존을 우선하였으나 조회 조건이 추가로 필요할 수 있습니다.

- 발주와 주문을 별도 도메인으로 분리하여 업무 흐름을 명확히 하였으나 중복 컬럼, 데이터가 생기고 일부 공통 처리 로직이 증가하는 구조가 되었습니다.

## 9. 아키텍처 개요

Next.js App Router 기반 구조로 화면과 API를 구성하였습니다.

관리자 기능은 /admin 경로 하위 모듈 단위로 분리하여 관리하도록 설계하였습니다.

인증은 Cookie 기반으로 처리하고 Middleware를 통해 보호 경로 접근을 제어합니다.

데이터 접근은 Prisma ORM을 활용한 서버 사이드 로직에서 수행합니다.
